# 개인정보 수집·보유·위탁 설계 (CLAW-15)

> 상태: 설계 확정 + 처리방침 초안. **최종 공개 전 개인정보 전문가 검토 필요** → 진행 중 유지.
> 이 설계는 클라이언트/서버 구현(CLAW-24 이벤트 스키마, CLAW-28 파기)이 반드시 준수한다.

## 1. 수집 허용목록 (allowlist)

이 목록에 없는 데이터는 수집하지 않는다. 클라이언트는 아래 외 필드를 이벤트에 담지 않는다.

- 클로애드 클라이언트 버전
- 운영체제 종류 (예: Windows/macOS/Linux, 세부 빌드 아님)
- Claude Code 실행 여부(불리언)
- 광고 표시 시간(시작/종료 timestamp), 광고 ID
- 가명화 머신 ID(로컬 생성 랜덤 키 해시)
- 세션 시작·종료 시간
- 클릭 여부
- 오류 로그(스택에 사용자 데이터 미포함)
- 회원 식별용: 이메일(또는 소셜 로그인 식별자), 동의 이력

## 2. 구조적 수집 금지 (denylist)

클라이언트 코드가 **접근 자체를 하지 않도록** 구현한다(정책 아닌 구조로 차단):

- 프롬프트 본문, Claude 응답 내용, 코드 내용
- 파일명·프로젝트 경로, Git 저장소명
- 터미널 명령어, 환경변수, 클립보드
- 회사명 추론 정보(레포·경로·코드로부터의 유추 포함)
- **하드웨어 식별자: MAC 주소, 디스크 시리얼, 하드웨어 UUID** (수집·전송 금지)

동시 노출 판정(CLAW-18 §6)에 필요한 최소 데이터만 수집한다: userId, 가명 machineId, startedAt/endedAt, sequence. 그 외 필드를 판정을 이유로 추가하지 않는다.

## 3. 동의 분리

| 동의 항목 | 필수/선택 | MVP |
|---|---|---|
| 서비스 이용약관 | 필수 | O |
| 개인정보 수집·이용 | 필수 | O |
| 광고성 정보 수신 | 선택 | O |
| 추가 텔레메트리 제공 | 선택 | O |
| 개인화 광고(사용자 선택 프로필 활용) | 별도 선택 | 프로필 기능은 P2 |
| 민감한 개발 맥락 활용 | — | **MVP 금지** |

가입 동의를 하나로 묶지 않는다. 각 동의는 독립 저장(consents 테이블, 버전·시각 기록).

## 4. 보유기간 (목적별 분리)

| 데이터 | 보유기간 |
|---|---|
| 회원 정보 | 탈퇴 시 지체 없이 파기, 단 법정 보관 항목 제외 |
| 광고 이벤트(노출·클릭) | 정산·분쟁 대응 기간 후 파기 |
| 부정사용 기록 | 차단·분쟁 대응 기간 |
| 지급·세무 기록 | 세법·전자상거래법상 보관기간(예: 5년) |
| 원본 IP 주소 | 수집 즉시 축약·가명처리(원본 미보관 원칙) |

정확한 연수는 관계 법령에 맞춰 전문가 검토로 확정한다.

## 5. 처리위탁 vs 제3자 제공

- **쿠폰 지급대행사**: 클로애드 지시에 따라 발송만 수행 → **처리위탁**으로 본다. 위탁계약서에 처리 목적·항목·기간·재위탁 제한·안전조치 명시.
- 공급사가 독립 목적으로 정보를 사용하면 제3자 제공에 해당 → 이 경우 별도 동의·고지 필요. **계약 단계에서 위탁 구조로 못박는다.**
- 위탁 시에도 처리방침에 수탁자·업무 내용을 공개한다.

## 6. 가명 머신 ID 규칙

- MAC 주소·디스크 시리얼·하드웨어 UUID 등 하드웨어 식별자를 수집·전송하지 않는다.
- 로컬에서 생성한 **랜덤 가명값**을 머신 식별자로 사용한다(참조 구현: `client/statusline.js`의 `getMachineId`, `crypto.randomBytes(16)`).
- 하드웨어 식별자를 쓰지 않으므로 "동일 기기를 완벽 식별한다"거나 "동일 지문 다계정을 확정 탐지한다"고 표현하지 않는다(CLAW-19).
- 서버는 이 가명 ID로 상한·동시노출·위험 신호만 판단하고, 실사용자 신원과 직접 연결하지 않는다(회원 매핑은 별도 최소 권한).

## 6.5 로그 마스킹

- 로그에 serveToken 전체 값, 인증 토큰, 이메일, 쿠폰 수신 정보 등을 남기지 않는다(필요 시 마스킹·해시).
- 원본 IP는 수집 즉시 축약·가명처리한다(§4).
