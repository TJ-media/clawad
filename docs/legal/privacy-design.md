# 개인정보 수집·보유·위탁 설계 (CLAW-15)

> ⚠️ **검토용 초안. 외부 공개 금지.** 미확정 플레이스홀더(§9)가 남아 있는 동안 이 문서와
> [privacy-policy-draft.md](privacy-policy-draft.md)를 실제 처리방침으로 게시하지 않는다.
> 최종 공개 전 개인정보 전문가 검토가 필요하다.
>
> 이 설계는 클라이언트/서버 구현(CLAW-24 이벤트 스키마, CLAW-28 파기)이 반드시 준수한다.
> 허용목록은 **실제 전송 스키마(§1)를 단일 기준**으로 하며, 문서가 코드보다 넓은 범위를 적지 않는다.

## 1. 수집 허용목록 (allowlist)

이 목록에 없는 데이터는 수집하지 않는다. 클라이언트는 아래 외 필드를 이벤트에 담지 않는다.

### 1.1 클라이언트가 실제로 전송하는 필드

`POST /v1/events` 본문 (참조 구현: `client/sync.js`) — **이 7개가 전부다.**

| 필드 | 내용 | 비고 |
|---|---|---|
| `serveToken` | 서버가 발급·서명한 불투명 토큰 | 클라이언트는 내용을 만들지 못한다 |
| `sequence` | 머신별 노출 순번(정수) | 멱등 키 재료 |
| `machineId` | 로컬 생성 랜덤 가명값 | 하드웨어 식별자 아님(§6) |
| `startedAt` / `endedAt` | **광고 표시** 시작·종료 timestamp | 세션 시각이 아니다 |
| `userId` | 인증된 사용자 식별자 | 서버가 세션·토큰으로 확정. 본문 자가신고값은 신뢰하지 않는다 |
| `clientVersion` | 클로애드 클라이언트 버전 | |

`POST /v1/machines` 본문: `userId`, `machineId`.

### 1.2 서버가 serveToken에서 추출해 저장하는 필드

클라이언트가 보내지 않는다. 서버가 토큰을 검증한 뒤 원장에 기록한다 (참조 구현: `server/index.js`의 `baseRec`).

- `tokenJti` — 토큰 `jti`. 멱등 키 재료
- `campaignId`, `campaignType` — 캠페인 식별자·유형(PAID/HOUSE/TEST)
- `idempotencyKey` = `SHA-256(tokenJti:machineId:sequence)`
- `decision`, `reason`, `confirmSeq`, `at`(서버 수신 시각)

> `creativeId`는 토큰 payload에 있으나 현재 원장에 저장하지 않는다. 저장을 시작하려면 이 목록을 먼저 갱신한다.

### 1.3 로컬에만 남고 전송하지 않는 값

`data/ledger.jsonl`, `data/state.json`, `data/machine.json` — 사용자 기기를 떠나지 않는다.

- `slotKey`, `adId`, `synced`(전송 여부 불리언), 로컬 기록 시각

### 1.4 조건부 항목 (기능 도입 시에만)

- **클릭 정보** — 클릭 가능한 인벤토리(CLAW-7)가 실제로 존재하고, 그에 대한 고지·동의가 적용되는 경우에만 수집한다. 현재 클릭 기능이 없으므로 **수집하지 않는다.**

### 1.5 회원 인증 (CLAW-22·CLAW-37)

- **소셜 로그인 식별자**: 공급자(Google·Kakao·Naver)와 그 공급자가 발급한 안정적 계정 식별자(`providerSubject`), 동의 이력.
- 공개 로그인은 소셜 전용이다. 소셜 사용자의 이메일은 저장하지 않으며(`users.email`은 NULL), 이메일·닉네임·프로필은 계정 식별자로 쓰지 않는다. (EMAIL·GitHub 식별자는 legacy로만 존재하며 신규 로그인에 쓰지 않는다.)
- 공급자 access/refresh 토큰과 authorization code는 로그인 검증에 필요한 동안만 메모리에 두고 DB·Redis에 장기 저장하지 않는다. `providerSubject` 외 추가 scope(이메일 등)는 목적·보유기간이 문서화되기 전 요청하지 않는다.
- 내부 세션 토큰: access 토큰은 클라이언트 메모리에만 둔다. refresh 토큰은 웹에서 JS가 읽을 수 없는 httpOnly+Secure 쿠키(Path=/v1/auth), CLI에서 `data/auth.json`(0600)로 두며, 서버에는 SHA-256 해시만 저장하고 사용 시 원자적으로 1회 소비·회전한다(CLAW-37·38·41). localStorage 등 JS 접근 저장소에 토큰을 두지 않는다.
- 내 정보 내보내기(`GET /v1/me/export`)는 연결된 provider·subject를 포함하되 토큰·비밀번호 해시는 제외한다. 탈퇴·파기 시 모든 identities를 제거한다(CLAW-28).

### 1.6 허용목록에서 제거한 항목 (실제 미수집)

아래는 이전 초안의 허용목록에 있었으나 **코드 어디에서도 수집하지 않는다.** 문서가 코드보다 넓은 범위를 선언하지 않도록 제거했다. 다시 추가하려면 수집 목적·보유기간·전송 여부를 정한 뒤 이 목록과 처리방침을 함께 갱신한다(AGENTS.md §"새 이벤트 필드").

- ~~운영체제 종류~~ — 전송 스키마에 없음
- ~~Claude Code 실행 여부(불리언)~~ — 수집·검증에 사용하지 않음
- ~~세션 시작·종료 시간~~ — `startedAt`/`endedAt`는 **광고 표시** 구간이지 세션 구간이 아니다
- ~~오류 로그~~ — 클라이언트가 서버로 오류를 전송하지 않음
- ~~`syncedAt`~~ — 전송하지 않는다. 로컬 `synced` 불리언과 서버 수신 시각 `at`으로 대체

## 2. 구조적 수집 금지 (denylist)

클라이언트 코드가 **접근 자체를 하지 않도록** 구현한다(정책 아닌 구조로 차단):

- 프롬프트 본문, Claude 응답 내용, 코드 내용
- 파일명·프로젝트 경로, Git 저장소명
- 터미널 명령어, 환경변수, 클립보드
- 회사명 추론 정보(레포·경로·코드로부터의 유추 포함)
- **하드웨어 식별자: MAC 주소, 디스크 시리얼, 하드웨어 UUID** (수집·전송 금지)
- **접속 IP** — 제품 이벤트 데이터로 수집하지 않는다(§6.6)

동시 노출 판정(CLAW-18 §6)에 필요한 최소 데이터만 수집한다: userId, 가명 machineId, startedAt/endedAt, sequence. 그 외 필드를 판정을 이유로 추가하지 않는다.

## 3. 동의 분리

| 동의 항목 | 필수/선택 | MVP |
|---|---|---|
| 서비스 이용약관 | 필수 | O |
| 개인정보 수집·이용 | 필수 | O |
| 광고성 정보 수신 | 선택 | O |
| 추가 텔레메트리 제공 | 선택 | O |
| 클릭 정보 수집 | 별도 선택 | 클릭 기능은 P2(CLAW-7). 기능 도입 전까지 미수집 |
| 개인화 광고(사용자 선택 프로필 활용) | 별도 선택 | 프로필 기능은 P2 |
| 민감한 개발 맥락 활용 | — | **MVP 금지** |

가입 동의를 하나로 묶지 않는다. 각 동의는 독립 저장(consents 테이블, 버전·시각 기록).

## 4. 보유기간 (목적별 분리)

| 데이터 | 보유기간 |
|---|---|
| 회원 정보 | 탈퇴 시 지체 없이 파기, 단 법정 보관 항목 제외 |
| 광고 이벤트(노출·클릭) | 정산·분쟁 대응 기간 후 파기 |
| 부정사용 기록 | 차단·분쟁 대응 기간 |
| 지급·세무 기록 | 세법·전자상거래법상 보관기간 `[미확정: 세법상 보관연수는 CLAW-13 세무 답변 필요]` |
| 접속 IP | 제품 이벤트로 **미수집**. 인프라 로그 한정 처리 → §6.6 |

정확한 연수는 관계 법령에 맞춰 전문가 검토로 확정한다. **연수를 임의로 확정해 코드·문서에 쓰지 않는다.**

## 5. 처리위탁 vs 제3자 제공

- **쿠폰 지급대행사**: 클로애드 지시에 따라 발송만 수행 → **처리위탁**으로 본다. 위탁계약서에 처리 목적·항목·기간·재위탁 제한·안전조치 명시.
- 공급사가 독립 목적으로 정보를 사용하면 제3자 제공에 해당 → 이 경우 별도 동의·고지 필요. **계약 단계에서 위탁 구조로 못박는다.**
- 위탁 시에도 처리방침에 수탁자·업무 내용을 공개한다.

## 6. 가명 머신 ID 규칙

- MAC 주소·디스크 시리얼·하드웨어 UUID 등 하드웨어 식별자를 수집·전송하지 않는다.
- 로컬에서 생성한 **랜덤 가명값**을 머신 식별자로 사용한다(참조 구현: `client/statusline.js`의 `getMachineId`, `crypto.randomBytes(16)`).
- 하드웨어 식별자를 쓰지 않으므로 "동일 기기를 완벽 식별한다"거나 "동일 지문 다계정을 확정 탐지한다"고 표현하지 않는다(CLAW-19).
- 서버는 이 가명 ID로 상한·동시노출·위험 신호만 판단하고, 실사용자 신원과 직접 연결하지 않는다(회원 매핑은 별도 최소 권한).

## 6.5 로그 마스킹

- 로그에 serveToken 전체 값, 인증 토큰, 이메일, 쿠폰 수신 정보 등을 남기지 않는다(필요 시 마스킹·해시).

## 6.6 접속 IP 처리 (제품 데이터 아님)

IP는 **제품 이벤트 데이터의 정식 수집 항목이 아니다.** 클라이언트는 IP를 전송하지 않으며, 서버 원장(`events.jsonl` / 운영 4원장)에 IP 컬럼을 두지 않는다.

**금지:**

- 광고 이벤트 DB에 IP를 저장하는 것
- IP를 타겟팅 신호로 사용하는 것
- IP로 고유 사용자·동일인·다계정을 판단하는 것 (동시 노출·상한 판정은 `userId` + 가명 `machineId`만 사용 — §2, CLAW-19)

**허용(불가피한 전송계층 처리):**

- 웹 서버·리버스 프록시·WAF·DDoS 방어 등 인프라·보안 로그에는 TCP 연결의 성격상 IP가 일시적으로 기록될 수 있다.
- 이 로그의 IP는 **기록 즉시 마스킹(예: 마지막 옥텟 절삭)하거나, 마스킹이 불가능한 보안 목적에 한해 짧은 보유기간 후 삭제**한다.
- 인프라 로그는 광고 이벤트 원장과 물리적·논리적으로 분리하며, 서로 조인하지 않는다.
- 보유기간·마스킹 방식은 `[미확정: 인프라 구성(CLAW-27) 확정 후 기입. 외부 공개 전 필수]`.

## 7. 산출물

- 처리방침 초안: [privacy-policy-draft.md](privacy-policy-draft.md)
- 구현 연계: CLAW-24(이벤트 스키마 필드 제한), CLAW-28(조회·탈퇴·파기), CLAW-27(접근통제·감사로그)

### 7.1 이용자 권리 구현 (CLAW-28, 구현됨)

- **조회·내보내기**: `GET /v1/me/export` — 수집 항목 전체를 기계 판독 JSON으로. 비밀번호 해시는 제외한다.
- **탈퇴·파기**: `DELETE /v1/me` — 즉시 서비스 이용 중단(기존 토큰·로그인 차단) + 직접 식별자 파기·가명화.
  - **append-only 원장(노출·리워드·과금)은 삭제하지 않는다.** userId는 가명 UUID이며, 세무·정산·분쟁 보관 의무가 있다. 대신 로그인 수단(identities)과 이메일을 파기해 신원 연결을 끊는다.
  - 미지급 **확정 리워드**가 있으면 탈퇴를 차단한다(`409 UNPAID_CONFIRMED_REWARDS`). 교환(CLAW-26) 후 탈퇴하거나 포기에 명시 동의해야 한다. 포기 시 반대 분개(ADMIN_ADJUST)로 잔액을 0으로 만든다.
  - 탈퇴 계정의 검증 중 리워드는 확정 배치에서 확정되지 않는다(신원 파기 계정에 확정잔액이 생기지 않게).
- **파기 로그**: `destruction_logs`(append-only). 파기된 값을 복제하지 않고 항목명·건수만 기록.
- **파기 배치**: `POST /internal/v1/privacy/run-retention-sweep`(SUPERADMIN) — 탈퇴 계정 잔여 식별자 정리. **보유기간 경과 원장 자동 파기는 세법상 보관연수 확정(CLAW-13) 후 확장한다.**
- **접속 IP**: 제품 이벤트로 수집하지 않으므로 파기 대상이 없다(§6.6). 인프라 로그의 IP 처리는 운영 절차([secrets-and-backup.md](../security/secrets-and-backup.md) §4).
