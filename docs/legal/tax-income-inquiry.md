# 사용자 리워드 소득 구분·원천징수 세무 질의서 (CLAW-13)

> 상태: **세무사 서면 답변 대기**. 이 문서는 세무 전문가에게 제시할 질의서와 내부 정리이며, 결론이 아니다.
> **원칙: 서면 답변 확보 전까지 코드·정책 문서에 세율·과세 기준을 확정 사실로 쓰지 않는다.**

## 1. 세무사에게 제시할 서비스 구조 설명

- 클로애드는 개발자 대상 광고 매체 플랫폼이다.
- 개발자(사용자)는 Claude Code 등 개발 도구 사용 중 상태줄에 표시되는 광고를 일정 시간(5초 이상) 노출한다.
- 서버가 검증한 유효 노출 수에 **비례하여 반복적으로** 비현금성 보상(모바일 쿠폰: 편의점 금액권·문화상품권 등)을 지급한다.
- 보상은 클로애드 내부 리워드 포인트로 적립되며, 사용자가 신청하면 외부 지급대행사를 통해 지정 상품(쿠폰)으로만 교환된다. 현금 출금·양도·충전은 없다.
- 지급 규모: 사용자당 월 수천~수만 원 수준 예상, 일부 헤비 유저는 더 높을 수 있음.

## 2. 확인 요청 항목

1. **소득 구분**: 위 보상이 세무상 ① 기타소득 ② 사업소득(인적용역성) ③ 단순 판촉성 리워드(과세 대상 아님/사업자 판촉비) 중 무엇에 해당하는가? 반복성·계속성이 판단에 미치는 영향은?
2. **원천징수 대상 여부·세율**: 원천징수 의무가 발생하는가? 발생 시 적용 세율(기타소득 22% vs 사업소득 3.3% 등)과 산정 방식(필요경비 인정 여부 포함)은?
3. **과세최저한**: "건별 5만원 이하 비과세"(기타소득 과세최저한)가 이 구조에 적용 가능한가? 동일인 합산·반복 지급 시에도 유효한가?
4. **현물(상품권) 지급**: 현금이 아닌 모바일 쿠폰으로 지급할 때 소득 평가액·필요경비·원천징수 처리는 현금과 어떻게 다른가?
5. **지급명세서·증빙**: 지급명세서(간이지급명세서 포함) 제출 의무와 주기, 사용자 주민등록번호 수집 필요 시점은?
6. **사업자 사용자(2단계 정산 모델)**: 사용자가 사업자등록번호를 제시하고 세금계산서를 발행하는 방식으로 정산할 경우의 처리 방법과, 이때 클로애드의 매입 처리는?

## 3. 내부 잠정 이해 (검증 대상 — 확정 아님)

- 반복성·계속성·성과 연동이 강하므로 **사업소득(인적용역)으로 해석될 여지**가 있다. 이 경우 원천징수는 3.3% 방식일 수 있고, "기타소득 22%·5만원 최저한"은 적용되지 않을 수 있다.
- 반대로 개별 이벤트성으로 보면 기타소득 가능성도 있다. **어느 쪽인지 서면 확인 없이 시스템을 확정하지 않는다.**
- 현물 지급이라도 경제적 가치가 있어 과세 대상 판단은 별개로 필요하다.

## 4. 답변이 설계에 미치는 영향 (분기)

| 답변 | 시스템 영향 |
|---|---|
| 사업소득 | 지급 원장에 사업소득 원천징수 필드, 연간 지급액 집계, 간이지급명세서 파이프라인 (CLAW-17·26) |
| 기타소득 | 과세최저한 판정 로직 + 동일인 연간 합산, 지급명세서 |
| 판촉성 비과세 | 원천징수 로직 최소화, 단 회계상 판촉비 처리 |
