# 무효 트래픽·리워드 회수·이의제기 정책 (CLAW-19)

> 상태: 확정 (v2). 검증 파이프라인(CLAW-6)·이용약관(CLAW-16)·QA(CLAW-29)가 근거로 삼는다.
> 핵심 변경: 다계정은 **확정 부정이 아니라 위험 신호**다. 동시 노출은 **제재가 아니라 중복 미인정**이다.

## 1. 확정 무효(자동 미과금·미적립) 사유

아래는 유효 노출로 인정하지 않는다. 자동으로 과금·리워드를 만들지 않는다:

- 토큰 검증 실패: `BAD_TOKEN`, `EXPIRED`
- viewability 미달·물리적으로 불가능한 간격: `BAD_INTERVAL`
- 순번 역행·토큰 재사용 이상: `SEQUENCE_ANOMALY`, `TOKEN_REUSE`
- 계정 단위 상한 초과: `OVER_CAP`
- 24시간 연속 등 비정상 장시간 무중단 패턴: `ABNORMAL_CONTINUOUS`
- 킬스위치 대상: `KILLED`

## 2. 동시 노출 — 중복 미인정 (제재 아님)

- 같은 사용자 계정의 여러 기기에서 시간상 겹친 노출은 **한 건만** 인정하고 나머지는 `CONCURRENT_USER_IMPRESSION`으로 거절한다.
- 이는 정상적인 다기기 사용에서 발생할 수 있으며 **부정행위가 아니다.** 사용자 제재 점수에 넣지 않고, 계정을 정지하지 않는다.
- 거절된 이벤트는 원장에 남기되 과금·리워드·유효 노출·리포트 유효 노출을 만들지 않는다. 수신 수·무효 사유 통계에는 포함한다.
- 판정·동시성 안전은 CLAW-18 §6 참조.

## 3. 다계정 — 위험 신호 (`MULTI_ACCOUNT_RISK`)

- **여러 사용자 계정을 만드는 행위 자체만으로 노출을 부정 처리하지 않는다.**
- 현재 시스템은 하드웨어 식별자를 수집하지 않으므로 동일인을 **확정적으로 식별할 수 없다.** "동일 지문 다계정을 확정 탐지한다"는 표현을 쓰지 않는다.
- 다계정·잦은 기기 교체는 `MULTI_ACCOUNT_RISK` 위험 신호로 기록하되, **단독으로 과금·리워드를 자동 취소하지 않는다.**
- 위험이 감지되면 **자동 제재보다 지급 보류 + 운영자 검토**를 우선한다.
- 가입 보너스·추천 보너스·쿠폰 교환 한도처럼 개인 단위 제한이 필요한 기능은 **향후 별도 본인확인 정책**으로 처리한다(현재 P0/P1에서 personId·개인 통합 식별 모델을 새로 만들지 않는다).

## 4. 판정 2단계

| 단계 | 시점 | 처리 |
|---|---|---|
| 실시간 필터 | 이벤트 수집 시(CLAW-6) | 토큰·viewability·동시노출·상한 등 명백한 사유는 즉시 판정 |
| 사후 배치 검수 | 리워드 확정 전 | 장시간 패턴 등 분석 → accrue_pending → 확정 또는 회수 |

## 5. 회수 규칙

- 사후 부정 판정 시: 광고주 크레딧 복원(billing.ivt_refund) + 사용자 리워드 회수(reward.claw_back). 원장에 사유 기록.
- 이미 교환·발송된 리워드는 회수 불가 → 다음 확정분 차감, 악의적 대량 부정은 계정 정지 후 미지급.

## 6. 지급 보류·이의제기

- 지급 보류: 위험 신호·부정 의심 사용자의 교환 신청은 보류(202) 후 운영자 검토. 해제 조건: 수동 검수 통과 또는 이의제기 인용.
- 이의제기: 접수 채널·처리 기한 안내, 판정 근거는 우회 방지를 위해 상세 룰 비공개(사유 코드·개괄만 안내), 인용 시 재적립(admin_adjust)·보류 해제.
- 재발 기준: 반복 위반 시 이용정지, 중대·고의는 해지(이용약관 근거).

## 7. 사유 코드 체계

`BAD_TOKEN`, `EXPIRED`, `BAD_INTERVAL`, `CONCURRENT_USER_IMPRESSION`(비제재), `OVER_CAP`, `SEQUENCE_ANOMALY`, `TOKEN_REUSE`, `ABNORMAL_CONTINUOUS`, `KILLED`, `MULTI_ACCOUNT_RISK`(위험 신호, 자동 취소 아님), `MANUAL_REVIEW`.

## 8. 백그라운드 세션 — 확정 사유로 쓰지 않음

- 현 statusLine 신호만으로 백그라운드 여부를 신뢰성 있게 판단할 수 없어 `BACKGROUND_SESSION`을 확정 거절 사유로 쓰지 않는다(CLAW-18 §8). 향후 신호로만 문서화한다.
