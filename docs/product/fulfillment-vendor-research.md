# 쿠폰 지급대행사 리서치 (CLAW-26 준비)

> 상태: 준비 리서치. **실제 지급대행 연동 착수는 세무(CLAW-13)·전자금융(CLAW-14) 서면 결론 후.**
> 이 문서는 벤더 선정·구조 설계를 위한 사전 조사이며, 특정 벤더 채택 결정이 아니다.

## 0. 왜 지금 이 문서만 쓰는가

CLAW-26(모바일 쿠폰 지급대행 연동)은 **실제 리워드 교환·발송**을 구현하는 이슈다. 그런데:

- 리워드의 소득 구분·원천징수 → **CLAW-13(세무사 서면 답변 대기)**
- 리워드 구조가 전자금융업 등록 대상인지 → **CLAW-14(법률 검토 대기)**
- 지급대행사가 "지급수단 발행자"가 아니라 클로애드의 **처리위탁 수탁자**임을 계약으로 못박는 구조 → CLAW-14·CLAW-15 §5

이 결론들이 벤더 API 계약·정산·세무 처리 방식을 좌우하므로, 결론 전에는 **연동 코드를 짓지 않고** 선정 기준과 구조만 정리한다(rules §5: 리워드 규칙, §0: P0 완료 전 P2 현금출금 금지).

## 1. 한국 리워드 앱은 어떻게 쿠폰을 지급하나

캐시워크·토스·오늘의집 등 리워드/포인트 앱은 **자체 쿠폰을 발행하지 않는다.** 포인트를 모으면
**B2B 모바일 쿠폰(기프티콘) 지급대행사**를 통해 스타벅스·편의점·문화상품권 등 **기존 브랜드 상품권**으로 교환해 준다.

즉 구조는:

```
사용자 포인트 적립 → 교환 신청 → 앱이 지급대행사 API 호출 → 대행사가 브랜드 쿠폰 발급·발송(카카오톡/SMS)
```

앱은 브랜드 상품권을 재고로 들고 있지 않고, 대행사가 실시간 발행·발송을 대행한다.

## 3. 왜 지급대행사를 쓰는가 (한국 맥락)

### 3.1 규제·법무 (클로애드에 가장 중요)

- **전자금융업 회피 구조**: 클로애드가 직접 범용 지급수단을 발행하면 선불전자지급수단·전자금융업 등록 대상이 될 수 있다(CLAW-14). 대행사가 브랜드 쿠폰을 발행·발송하고 클로애드는 **광고 매체 사업자로서 지급을 위탁**하면, 클로애드는 지급수단 발행자가 아니게 된다. 이 구조를 **계약 단계에서 처리위탁으로 못박는다**(privacy-design.md §5).
- **처리위탁 구조**: 대행사는 클로애드 지시로 발송만 수행 → 개인정보 처리위탁(제3자 제공 아님). 위탁계약서에 목적·항목·기간·재위탁 제한·안전조치 명시.

### 3.2 운영·비용

- **재고 부담 없음**: 브랜드 상품권을 직접 매입·보관하지 않는다. 대행사가 실시간 발행.
- **인쇄·물류 비용 제거**: 카카오톡·SMS·이메일 등 디지털 발송이라 실물 배송비가 없다.
- **자동화**: API로 "특정 행동(교환 신청) 발생 시 자동 발급" 연동. 취소·재전송 지원.
- **정산 편의**: 디지털 처리로 발송·사용·정산이 자동 집계.
- **브랜드 다양성**: 다수 브랜드 상품권을 한 API로 취급 → 사용자 선택권 확보.

### 3.3 세무 (CLAW-13 확정 전까지 미확정)

- 모바일 쿠폰(기프티콘)은 일반적으로 **비과세 유가증권**으로 분류되어 세금계산서 대신 **전자거래명세서·견적서**로 증빙하는 경우가 있다(대행사 안내 기준).
- **단, 사용자에게 지급되는 리워드의 소득 성격·원천징수는 별개 문제**이며 CLAW-13 서면 답변 전까지 확정 사실로 쓰지 않는다. 대행사 매입 증빙과 사용자 지급 소득 처리는 구분해서 검토한다.

## 4. 연동 시 클로애드가 지켜야 할 설계 원칙

CLAW-26 실제 연동 때 반영할 요건(지금은 목록만):

- **화이트리스트 상품 카탈로그**: 지정 상품(모바일 쿠폰)만 교환. 임의 현금·범용 지급수단 금지(rules §5).
- **지급 원장(redemption_ledger, CLAW-17)**: `request → supplier_order → delivered / delivery_failed / canceled / resent` 상태 전이. append-only.
- **reward_ledger REDEEM_DEBIT 연결**: 교환 신청 시 포인트 차감(이미 enum 정의됨, CLAW-5). 차감과 발주는 멱등·원자적으로.
- **세무 상태 필드**: 지급 원장에 `tax_status`(none/withholding_pending/…)만 두고 세율·기준은 하드코딩하지 않는다(CLAW-13 결론 반영).
- **비밀정보**: 대행사 API 키는 시크릿 매니저로만(코드·레포 반입 금지, CLAW-27).
- **개인정보**: 쿠폰 수신정보(전화번호 등)는 발송 목적 최소 수집·처리위탁, 로그 마스킹(privacy-design.md §6.5).
- **실패 처리**: 발급 실패 시 포인트 원복(반대 분개), 재전송·취소 경로.

## 5. 알파 교환 임계값

- 현재 정책 `minimumRedemptionPoints = 3000P`(= 3,000원 상당). 상한 사용자 기준 20일에 첫 교환.
- 사용자 결정: **3,000P부터 교환 가능**으로 유지한다(빠른 첫 보상 지향). 정책값이라 코드 변경 없이 조정 가능.
- 참고: 캐시워크도 만보기 월 3,000원 수준에서 첫 기프티콘 교환. 유사한 눈높이.
- 실제 "교환 가능" 기능(포인트 차감 + 쿠폰 발급)은 CLAW-26 본구현 = 세무·전자금융 결론 후.

## 6. 다음 단계 (게이트)

1. CLAW-13 세무 서면 답변 → 리워드 소득 처리·지급 원장 세무 필드 확정
2. CLAW-14 전자금융거래법 검토 → 처리위탁 구조·리워드 규모 상한 확정
3. 위 둘 확정 후 → 지급대행사 1~2곳 실사·계약 → CLAW-26 연동 구현(지급 원장·API 클라이언트·교환 UI)
