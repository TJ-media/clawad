# Clawad Rules (v2)

이 문서는 클로애드 코드 작성 규칙을 정의한다.
우선순위: **[CRITICAL]** 위반 시 머지 불가 / **[HIGH]** 리뷰에서 반드시 지적 / **[MEDIUM]** 가능하면 수정

---

## 1. 클린룸 원칙 [CRITICAL]

- kickbacks.ai 원본 레포 코드를 열람·인용·복제하지 않는다. 공개 제품 설명(아이디어)만 참고한다.
- "kickback" 계열 네이밍을 코드·문서·브랜드 어디에도 쓰지 않는다.

## 2. 클라이언트 보안 경계 [CRITICAL]

클라이언트는 사용자가 완전히 통제하는 환경이다. 다음 값은 클라이언트가 **절대 결정·계산·전송하지 않는다**:

- 노출 단가, 사용자 배분율, 최종 리워드 금액, 광고주 차감 금액
- 캠페인 활성 여부, 일일 적립 상한, 부정 여부, 회원 잔액

클라이언트는 **사실만** 보고한다: serveToken, startedAt/endedAt, sequence, machineId(가명), clientVersion.
금액은 서버가 캠페인 계약과 정책으로 계산한다. 클라이언트 화면의 수익 표시는 "예상"임을 명시한다.

## 3. 핫패스 규칙 (statusline) [CRITICAL]

- 네트워크 호출 금지. 로컬 캐시(sync 데몬이 프리페치)만 읽는다.
- stdin이 비어 있거나 깨진 JSON이어도 반드시 광고 한 줄을 출력하고 exit 0 한다.
- 출력은 정확히 한 줄. 광고 문구에 `[광고]` 표기 필수(표시광고법) — 제거·변경 금지.
- Claude Code 공식 메시지·오류 메시지로 오인될 문구 렌더링 금지.

## 4. 집계·원장 무결성 [CRITICAL]

- 노출 인정은 **서버 검증**(serveToken 서명·만료·순번·간격·상한)을 통과한 것만. 클라이언트 자기 신고를 신뢰하지 않는다.
- **멱등 키는 서버가 생성한다**: `SHA-256(tokenJti + ":" + machineId + ":" + sequence)`. serveToken에 `jti`를 담는다. 클라이언트는 HMAC이나 서비스 비밀 키를 보유하지 않는다. DB `UNIQUE(token_jti, machine_id, sequence)`로 중복 차단.
- 원장은 4종 분리(광고 이벤트/광고주 과금/사용자 리워드/지급), 전부 append-only. 정정은 반대 분개 또는 상태 전이 기록으로만.
- balance 필드를 직접 수정하는 코드 금지 — 잔액은 항상 원장 합산(또는 원장 기반 캐시).
- 예산 차감: 서빙 시 예약 → 검증 통과 시 확정 → 거절/**토큰 만료 시 예약 해제(멱등)**. 행 잠금/원자적 업데이트 필수.

## 4b. 계정·기기·동시 노출 [CRITICAL]

- 한 계정에 활성 기기 **최대 N대**(정책값, 기본 3). 초과 등록은 `409 MACHINE_LIMIT_EXCEEDED`. 등록 검사는 DB 트랜잭션 안에서.
- 여러 로그인 수단은 한 계정에 연결. 여러 계정으로 보지 않는다. **다계정 자체를 자동 차단·부정 처리하지 않는다** — 위험 신호(`MULTI_ACCOUNT_RISK`)로만 다루고 자동 제재보다 지급 보류·운영자 검토 우선.
- 상한·빈도·리워드 상한은 **기기별이 아니라 사용자 계정 단위**로 적용.
- 같은 계정의 여러 기기 동시 노출은 **한 건만 인정**(`CONCURRENT_USER_IMPRESSION`). 제재가 아니라 중복 미인정. 나머지는 과금·리워드·유효 노출을 만들지 않되 원장·수신 통계에는 남긴다. 동시성은 PostgreSQL 트랜잭션/잠금으로 보장(메모리 캐시만 금지).
- 백그라운드 세션은 현 신호로 확정 판단 불가 → `BACKGROUND_SESSION`을 확정 거절 사유로 쓰지 않는다.

## 5. 리워드·세무·캠페인 규칙 [CRITICAL]

- 모든 리워드 단가·상한·간격은 서버 정책 설정(`policy/reward-policy.default.json`, 운영은 정책 테이블)에서만 관리한다. **코드에 숫자 하드코딩 금지.** 값 변경은 `policy.js`의 불변식(일일 상한 ≤ 최대 적립, 최소 교환 도달일 ≤ 허용일)을 통과해야 한다.
- 캠페인 유형 **PAID/HOUSE/TEST**의 과금·리워드 자격을 강제한다(`server/lib/campaign.js`). HOUSE·TEST는 광고주 매출·미지급 리워드 부채를 만들지 않는다.
- 리워드는 비구매형·비양도형: 충전·양도·사용자 간 이전·현금 환급 기능 구현 금지 (P2 현금 출금은 CLAW-13·14 결론 후).
- "기타소득 22%", "건별 5만원 비과세" 등 세율·과세 기준을 코드·문서에 확정 사실로 쓰지 않는다. 세무 상태 필드만 둔다.

## 6. 프라이버시 [CRITICAL]

- **수집 허용목록의 단일 기준은 실제 전송 스키마**다 (`docs/legal/privacy-design.md` §1). 문서가 코드보다 넓은 범위를 선언하지 않는다.
- 클라이언트가 전송하는 필드는 다음 7개뿐이다: `serveToken`, `sequence`, `machineId`, `startedAt`, `endedAt`, `userId`, `clientVersion`. 이 외 필드를 이벤트에 추가하지 않는다.
- `campaignId`·`tokenJti`는 클라이언트가 보내지 않는다. 서버가 serveToken을 검증해 추출·저장한다.
- `startedAt`/`endedAt`는 **광고 표시 구간**이다. 세션 시작·종료 시간을 수집하지 않는다.
- OS 종류·Claude Code 실행 여부·오류 로그는 **수집하지 않는다**. 클릭 정보는 클릭 기능과 별도 동의가 있을 때만 수집한다.
- **접속 IP는 제품 이벤트 데이터가 아니다.** 이벤트 원장에 IP 컬럼을 두지 않고, 타겟팅·고유 사용자·다계정 판단에 사용하지 않는다. 인프라·보안 로그의 일시적 기록은 즉시 마스킹하거나 짧은 보유 후 삭제하며, 이벤트 원장과 조인하지 않는다.
- 수집 금지(코드 자체가 접근하지 않아야 함): 프롬프트 본문, Claude 응답, 코드 내용, 파일명·경로, Git 레포명, 터미널 명령어, 환경변수, 클립보드, 회사명 추론 정보.
- 머신 ID는 로컬 생성 랜덤 키 기반 가명 식별자 — MAC·디스크 시리얼 등 하드웨어 식별자 전송 금지.
- 새 이벤트 필드는 수집 목적·보유기간·전송 여부를 정하고 `privacy-design.md` §1과 처리방침을 함께 갱신한 뒤 추가한다.

## 7. 설치 UX [HIGH]

- 설치 전 변경 내용 고지, 기존 statusLine 설정 백업, uninstall 시 원상복구.
- 광고 일시중지·빈도 제어를 사용자에게 제공. 서버 킬스위치 유지.
- "Claude 공식 서비스"로 오인될 표현 금지.

## 8. 의존성·견고성 [HIGH]

- 클라이언트(client-cli)는 node 내장 모듈만 사용. 서버(NestJS 전환 후)는 승인된 패키지만.
- 모든 JSON 파싱 전에 BOM(U+FEFF) 제거. 파일 부재·손상 시 fallback 동작.
- 서버 API는 잘못된 입력에 4xx JSON 응답. 크래시 금지.

## 9. 스타일 [MEDIUM]

- 함수·변수 camelCase, 상수 UPPER_SNAKE_CASE. 주석·사용자 문자열은 한국어.
- 리워드 상태 용어 통일: 예상 적립 / 검증 중 / 확정 리워드 / 교환 대기 / 지급 완료 / 회수.

## 10. 금지 사항 요약

- [CRITICAL] 클라이언트의 금액 계산·전송 (gross/user 류 필드), 클라이언트 HMAC/비밀 키 보유
- [CRITICAL] 서버 검증 없는 노출 인정 / 클라이언트가 멱등 키를 만드는 구조
- [CRITICAL] balance 직접 수정 / 원장 항목 수정·삭제
- [CRITICAL] 리워드 단가·상한·간격 하드코딩(정책 설정 외), 세율·과세 기준 하드코딩
- [CRITICAL] HOUSE·TEST 광고가 광고주 매출·미지급 리워드 부채 발생
- [CRITICAL] 다계정 자동 차단/자동 부정 처리, 하드웨어 식별자 수집
- [CRITICAL] 수집 금지 목록 데이터 접근 / 원본 코드 참조
- [HIGH] `[광고]` 표기 제거, 동시 노출을 제재로 처리
